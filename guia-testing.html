Anatomia de un Unit Test

Patron AAA: Arrange - Act - Assert

@Test
void execute_shouldCreateTodoWithCorrectFields() {
    // ARRANGE - preparar datos
    CreateTodoDto dto = new CreateTodoDto(
        "Mi tarea", "Descripcion de la tarea"
    );

    // ACT - ejecutar lo que queremos probar
    Todo result = createTodoUseCase.execute(dto);

    // ASSERT - verificar resultados
    assertNotNull(result);
    assertEquals("Mi tarea", result.getTitle());
    assertFalse(result.isCompleted());
}
Arrange
Preparar objetos, mocks y datos de entrada
Act
Ejecutar el metodo que se esta probando
Assert
Verificar que el resultado es el esperado

Mockito: Simulando Dependencias

Aislamos el codigo bajo prueba

@BeforeEach
void setUp() {
    // Crear mocks de las dependencias
    todoRepository = mock(TodoRepository.class);
    authenticatedUserContext = mock(AuthenticatedUserContext.class);

    // Definir comportamiento del mock
    when(authenticatedUserContext.getCurrentUser())
        .thenReturn(currentUser);

    when(todoRepository.save(any(Todo.class)))
        .thenAnswer(inv -> inv.getArgument(0));

    // Inyectar mocks manualmente
    createTodoUseCase = new CreateTodoUseCase(
        todoRepository, authenticatedUserContext
    );
}
Concepto clave: El mock no tiene logica real. Solo devuelve lo que le dices con when().thenReturn(). Esto aisla tu test de la base de datos, Firebase, etc.

Verify y ArgumentCaptor

Verificar que se llamaron los metodos correctos

@Test
void execute_shouldCallRepositorySave() {
    CreateTodoDto dto = new CreateTodoDto("Tarea", "Desc");

    createTodoUseCase.execute(dto);

    // Capturar el argumento que se paso a save()
    ArgumentCaptor<Todo> captor =
        ArgumentCaptor.forClass(Todo.class);

    // Verificar que save() se llamo exactamente 1 vez
    verify(todoRepository, times(1)).save(captor.capture());

    // Inspeccionar que se guardo
    Todo saved = captor.getValue();
    assertEquals("Tarea", saved.getTitle());
}
verify()
Confirma que un metodo del mock fue invocado N veces
ArgumentCaptor
Captura el argumento real que se paso al mock

Integration Test con @QuarkusTest

Probando endpoints HTTP reales

@QuarkusTest
class TodoResourceTest {

    @InjectMock
    CreateTodoUseCase createTodoUseCase;

    @Test
    void createTodo_shouldReturn200WithTodo() {
        // Mock del use case
        when(createTodoUseCase.execute(any()))
            .thenReturn(mockTodo);

        // Llamada HTTP real al servidor
        given()
            .contentType("application/json")
            .header("Authorization", "Bearer mock-token")
            .body("{\"title\":\"Mi tarea\"}")
        .when()
            .post("/todo")
        .then()
            .statusCode(200)
            .body("title", equalTo("Mi tarea"));
    }
}
@InjectMock reemplaza el bean real de Quarkus con un mock de Mockito dentro del contexto CDI.

Mock del Auth Filter

Simulando autenticacion de Firebase en tests

Problema

En produccion, FirebaseAuthFilter valida tokens reales de Firebase. En tests no tenemos Firebase.

⬇

Solucion: MockFirebaseAuthFilter

Un filtro simplificado que solo checa si hay header Authorization: Bearer *

// Si no hay token, rechazar
if (authHeader == null || !authHeader.startsWith("Bearer ")) {
    requestContext.abortWith(Response.status(401).build());
    return;
}

// Si hay token, crear usuario mock
CurrentUser currentUser = new CurrentUser(
    mockUUID, "mock-firebase-uid",
    "test@test.com", "USER", "Test User"
);

Testeando Seguridad

Verificar que endpoints protegidos requieren token

✅ Con token

given()
  .header("Authorization",
         "Bearer mock-token")
.when()
  .get("/todo/test")
.then()
  .statusCode(200);

❌ Sin token

given()
.when()
  .get("/todo/test")
.then()
  .statusCode(401);
Siempre prueba ambos escenarios: acceso autorizado Y no autorizado. Un endpoint sin test de 401 es un riesgo de seguridad no verificado.

Resumen / Checklist

Lo que debes recordar

  • ✓Patron AAA — Arrange, Act, Assert en cada test
  • ✓Mockito para aislar dependencias (DB, Firebase, etc.)
  • ✓@QuarkusTest levanta el servidor para integration tests
  • ✓@InjectMock reemplaza beans reales en el contexto CDI
  • ✓REST Assured para probar endpoints HTTP con given/when/then
  • ✓verify() para confirmar que se invocaron metodos del mock
  • ✓Probar seguridad — siempre testear con y sin token
  • ✓Naming descriptivo: metodo_shouldBehavior
